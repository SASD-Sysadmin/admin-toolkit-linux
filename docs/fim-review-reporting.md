# FIM Review Reporting

Roadmap Milestone 5 adds a focused File Integrity Monitoring review workflow.
It is intentionally small and transparent: create a baseline, compare a later
state against a baseline, and summarize the result for human review.

## Scope

The FIM tools are useful for lab systems, small servers and learning-oriented
administration work. They are not a replacement for an enterprise FIM platform,
EDR agent, SIEM correlation rule set or immutable audit store.

The scripts are read-only for monitored files. They may write generated baseline
and report files below the selected output directory.

## Main workflow

Create a focused review using the default baseline paths:

```bash
scripts/reporting/sasd-run-fim-review.sh --output ./reports/fim-current
```

Create a baseline for selected paths:

```bash
scripts/reporting/sasd-run-fim-review.sh \
  --path /etc/ssh \
  --path /etc/systemd \
  --output ./reports/fim-etc-baseline
```

Check a later system state against a stored baseline:

```bash
scripts/reporting/sasd-run-fim-review.sh \
  --baseline ./reports/fim-etc-baseline/01-current-baseline.tsv \
  --output ./reports/fim-etc-check
```

## Generated files

| File | Purpose |
| --- | --- |
| `INDEX.md` | Collection index and review order. |
| `status.tsv` | Exit status per command. |
| `01-current-baseline.tsv` | Fresh baseline generated during the run. |
| `02-fim-check.md` | Check result when an existing baseline is provided. |
| `03-fim-report.md` | Human-friendly summary generated by `sasd-fim-report.py`. |

## Exit statuses

| Status | Meaning |
| ---: | --- |
| `0` | Baseline/check/report generation completed without findings. |
| `1` | FIM check/report found changed, missing or unreadable entries. |
| `3` | Invalid input, unreadable baseline, missing prerequisite or inconclusive report. |

A status of `1` is a review signal, not a shell failure. It means the baseline
and current system state differ in at least one important way.

## Security notes

Baselines and reports can reveal sensitive information:

- file paths
- ownership and group names
- permissions
- sizes
- modification timestamps
- cryptographic hashes

Do not publish real FIM baselines or reports without review. Store long-lived
baselines with restrictive permissions and treat them as operational metadata.

## Review guidance

Changed entries are not automatically malicious. They may reflect expected
package updates, configuration changes, account changes or service deployment.
The important part is that the difference becomes visible and reviewable.

Good review questions:

- Was a change expected?
- Is the owner/group/mode still correct?
- Did a monitored file disappear?
- Did a file become unreadable because permissions changed?
- Should the baseline be refreshed after a confirmed legitimate change?
